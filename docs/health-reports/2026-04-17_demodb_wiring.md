## What was built

DemoDB is now a fully wired, live database that mirrors WorkingDB's schema.
Demo tenants (`Tenant.status = 'DEMO'`) have their CRM data isolated to demodb
instead of polluting production WorkingDB.

---

## Key decisions

| Item | Decision | Reason |
|------|----------|--------|
| Schema approach | **Approach B (copy)** — not symlinks | Prisma doesn't resolve symlinks for enum references across schema folders |
| subscriptionType | **Added `DEMO` to `TenantStatus` enum** | Existing field, no new column needed |
| Routing | **`PrismaService.getWorkingOrDemoClient(tenantId)`** | Simpler than middleware; avoids refactoring 100+ service files |
| Accessor name | **`prisma.demo` = DemoDB client** | Reclaimed from backward-compat `get demo()` (WorkingDB model accessor) |
| Backward-compat fix | **1 file updated** (`audit-snapshot.service.ts:48`) | Changed `prisma.demo` → `prisma.working.demo` |

## Sync workflow

After any `prisma/working/v1/` schema change:
```bash
npm run sync:demo-schema     # updates demo/v1/ domain files + enums
npm run prisma:generate      # regenerate all clients including demo
npx prisma db push --schema=prisma/demo/v1  # sync schema to demodb
```
